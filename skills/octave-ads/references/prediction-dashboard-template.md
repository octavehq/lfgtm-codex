# Prediction Dashboard HTML Template

A self-contained HTML dashboard that visualizes the prediction cards file (`~/.octave/predictions/<MCC_ID>.json`) — showing active predictions, evaluated outcomes, calibration stats, and the lessons log. This is the tracker that makes the "AI tool with a verifiable track record" idea real: a user can open this in their browser, scan what's pending / confirmed / refuted, see the hit rate per prediction type, and read the accumulated lessons.

This is the visual analog of `resonance-report-template.md`, but focused on the prediction history rather than a single run's findings.

## When to generate

Offer as a follow-up action alongside the resonance report, or any time the user asks for a "prediction dashboard", "track record", "calibration view", or "show me the prediction file". Can also be auto-generated at the end of every resonance loop run if the user wants persistent tracking they can bookmark.

## File output

Save to `~/Desktop/prediction-dashboard-<workspace-slug>-<YYYY-MM-DD>.html`. Overwrite any existing file at that path (the dashboard is always a snapshot of the current state, not an archive of prior states).

## Required sections (in order)

1. **Header band** — account name, MCC ID, schema version, last-evaluated-at timestamp, summary stats (N total, N evaluated, N pending, N tentative)

2. **Calibration summary cards** — 4-6 large cards showing the headline calibration numbers: directional hit rate, CONFIRMED count, REFUTED count, INCONCLUSIVE counts split by FAVORABLE/UNFAVORABLE, tentative count. Color-coded.

3. **Hit rate by prediction type** — horizontal bar chart (inline SVG) showing each prediction type's hit rate with sample size. Types with N<10 are shown in gray ("insufficient data for calibration"); types with hit rate >80% in green, 30-80% in amber, <30% in red.

4. **Active predictions (pending + tentative)** — a card per prediction, grouped by evaluation date. Each card shows: claim, type, confidence, evaluation window, days until re-evaluation, and the partial-evaluation snapshot if any. Tentative predictions get a visual "TENTATIVE" indicator.

5. **Resolved predictions (chronological, most recent first)** — a card per resolved prediction. Each card shows: claim, status (color-coded), evaluated-against data, evaluation notes, and a link to the corresponding new prediction if one was generated to replace it.

6. **Lessons log** — the `calibration.lessons` array rendered as a numbered list. Each lesson is a card with the full text and a small tag showing which backtest or run discovered it.

7. **Footer** — link to `~/.octave/predictions/<MCC_ID>.json` path, generation timestamp, "generated by /octave:ads resonance"

## Visual design

Same palette and typography as `resonance-report-template.md`:

- **Typography**: Inter for body, JetBrains Mono for metrics
- **Palette**:
  - Background: `#FAFAF9`
  - Text: `#0A0A0A`
  - CONFIRMED: `#16A34A` (green)
  - REFUTED: `#DC2626` (red)
  - INCONCLUSIVE_FAVORABLE: `#84CC16` (lime)
  - INCONCLUSIVE_UNFAVORABLE: `#F59E0B` (amber)
  - PENDING: `#6B7280` (gray)
  - TENTATIVE: `#8B5CF6` (purple) with diagonal stripe overlay
- **Hit rate bar chart**: inline SVG, ~30 lines, one `<rect>` per prediction type, text labels inline

The dashboard should feel like a modest internal tool — dense but not crowded, scannable by someone who just glanced at it, printable if needed.

## Implementation notes

- **Data source**: read `~/.octave/predictions/<MCC_ID>.json` directly. Every field in the dashboard maps to a field in the JSON. The LLM's job is to render it cleanly, nothing more.
- **Tentative rendering**: use the diagonal stripe background pattern on `.prediction-card.tentative` to make the status visually distinct from non-tentative cards. Users should be able to see "this CONFIRMED is tentative" at a glance.
- **Hit rate classification**: 
  - `>= 80%` and `N >= 10` → `.high` (green)
  - `30-80%` and `N >= 10` → `.mid` (amber)
  - `< 30%` and `N >= 10` → `.low` (red)
  - `N < 10` → `.insufficient` (gray) regardless of hit rate — small sample is not a basis for tuning
- **Directional hit rate** = `(confirmed + inconclusive_favorable) / (total_resolved)`. This is the softer measure that includes cases where the volume gate failed but the directional signal supported the prediction.
- **Grouping in "Active predictions"**: sort by `evaluation_window.end` ascending so the user sees what resolves next.
- **Empty states**: if `predictions` is empty, show a friendly "No predictions yet. Run /octave:ads resonance to generate the first set." message instead of blank sections.
- **Color accessibility**: ensure status badges have sufficient contrast on their background colors. Green/red should be distinguishable by text labels in addition to color for colorblind users.
- **The path link in the header** uses `file://` protocol so clicking it opens the raw JSON in the user's default text editor.
